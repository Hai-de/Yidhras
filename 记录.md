## 当前进行中：Ownership Matrix 审计

### 审计对象与当前观察

#### 1. `Event`

**当前宿主：** kernel-side Prisma  
**主要使用面：**

- `domain/rule/enforcement_engine.ts`
  - objective enforcement 在 `emit_events` 时直接写 `Prisma Event`
- `packs/runtime/projections/narrative_projection_service.ts`
  - narrative timeline 从 `context.prisma.event.findMany(...)` 读取事件
- `app/services/audit.ts`
  - unified audit feed 使用 `Event` 作为审计项
- `app/services/inference_workflow/repository.ts`
  - 作为 workflow follow-up signal 的输入来源
- `memory/short_term_adapter.ts`
  - 作为短期记忆输入之一

**初步结论：** `bridge-between-both`  
**理由：**

- 事件的产生源头可以来自 pack-governance objective rule
- 但事件的消费面明显跨越：audit / memory / workflow follow-up / narrative projection
- 它不是纯 pack 内部 runtime 状态，而是跨 pack-governance 与 kernel observability 的共享证据对象

**建议：**

- 短期继续保留在 kernel-side Prisma
- 在 design / ARCH / API 中明确其角色：
  - **kernel-hosted shared evidence/event ledger**
  - 而不是 pack-owned narrative source of truth
- 若未来引入 `PackOutboxEvent`，也应把 `Event` 视为投影/证据层宿主，而非直接迁入 pack runtime

---

#### 2. `Post`

**当前宿主：** kernel-side Prisma  
**主要使用面：**

- `app/services/social.ts`
  - 社交流主数据与写入宿主
- `app/services/audit.ts`
  - post 被纳入统一 audit feed
- `memory/short_term_adapter.ts`
  - 作为短期记忆输入之一

**初步结论：** `retain-in-kernel`  
**理由：**

- `Post` 本质上属于 social layer / public communication surface
- 它并不是 world governance core 的内生状态对象
- 它天然需要统一的身份、权限、审核、审计与 feed 查询能力
- 迁入 pack runtime 会削弱 social layer 的统一宿主意义

**建议：**

- 明确长期保留在 kernel-side Prisma
- 在 ownership matrix 中将其标记为：
  - **social/public communication kernel object**
- pack-governance 若影响社交流，只通过 invocation / audit / event / feed side-effect 间接体现，不将 `Post` pack-owned 化

---

#### 3. `ActionIntent`

**当前宿主：** kernel-side Prisma  
**主要使用面：**

- `app/services/action_dispatcher.ts`
  - 调度、claim、dispatch、drop、status 更新主宿主
- `domain/invocation/invocation_dispatcher.ts`
  - 将 `ActionIntent` bridge 为 `InvocationRequest`
- `app/services/audit.ts`
  - workflow 视图与意图轨迹
- `app/services/relational/graph_projection.ts`
  - graph relay/container 结构的一部分
- `memory/short_term_adapter.ts`
  - 短期记忆输入
- `inference/sinks/prisma.ts`
  - inference 输出持久化为 action intent

**初步结论：** `retain-in-kernel`  
**理由：**

- `ActionIntent` 是 workflow dispatch / reliability / retry / transmission policy 的核心对象
- 它是 subject-side / workflow-side 结构，而不是 world objective state 本身
- 当前 pack-governance 只是消费 `ActionIntent` 并桥接为 `InvocationRequest`
- 因此它更像 kernel workflow envelope，而不是 pack runtime object

**建议：**

- 长期保留在 kernel-side Prisma
- 明确其角色为：
  - **workflow / dispatch envelope**
  - 与 pack-governance 的关系是 bridge，而非宿主
- 后续若要进一步纯化，应强化 “ActionIntent -> InvocationRequest” 边界，而不是把 `ActionIntent` 迁入 pack runtime

## 对 `PackOutboxEvent` 的判断

**当前代码中尚无正式 `PackOutboxEvent` 模型。**

### 初步结论

- **本轮不引入 `PackOutboxEvent` 实现**
- 但应在文档中明确：
  - 当前 pack-governance 与 kernel evidence 的桥，主要通过：
    - objective enforcement 直接写 `Event`
    - projection extraction / audit aggregation
    - workflow bridge

### 不立即引入的理由

1. 当前 bridge 已足够支撑 narrative / audit / operator / memory / workflow follow-up
2. 现阶段更重要的是先把 ownership matrix 说清楚
3. 若过早引入 `PackOutboxEvent`，容易把“边界澄清问题”提前升级成“事件基础设施重构问题”

### 后续触发条件

若未来出现以下任一需求，可重新考虑：

- pack runtime 需要更严格的异步出站事件边界
- 多 pack / 多 runtime instance 需要稳定的 outbox/extractor contract
- narrative/event audit 需要更强的幂等同步与重放机制

---

## 结论

ownership matrix 审计显示：

- **world governance core** 继续 pack-owned
- **workflow / social / audit / relational evidence** 大部分应继续保留 kernel-side
- 真正跨边界的核心对象主要是：
  - `Event`
  - 以及 `ActionIntent -> InvocationRequest` 这条 bridge

下一步建议：

1. 把上述决策同步到 design / ARCH / API
2. 固化 `/api/access-policy/*` 作为独立 access-policy 子系统的位置，并继续保持 `/api/entities/:id/overview` 为唯一 canonical entity overview
3. 再处理 compat API 的剩余长期调整
