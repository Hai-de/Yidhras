## Requirement Alignment Notes (2026-03-21)

Purpose / 用途:

- Record scope/acceptance items that were previously implicit during smoke-test rollout.
- 记录冒烟测试落地过程中此前未显式定义的验收边界，避免后续口径漂移。

### A1. Degraded Startup Acceptance / 降级启动验收

- Decision / 决策:
  - `health_level=degraded` and `runtime_ready=false` is treated as acceptable startup state for first-time clones.
  - 首次拉取仓库时内容可能为空，降级启动视为通过，不作为冒烟失败条件。
- Enforcement / 落地:
  - Startup and endpoint smoke scripts accept degraded path and validate structured responses in both branches.

### A2. Runtime-Not-Ready API Consistency / 运行时未就绪接口一致性

- Problem / 问题:
  - `/api/clock/formatted` previously returned `500/CLOCK_FORMAT_ERR` when runtime was not ready, inconsistent with other world-pack-gated endpoints.
- Decision / 决策:
  - Runtime-not-ready path must return `503/WORLD_PACK_NOT_READY` consistently.
- Enforcement / 落地:
  - Route behavior aligned in server entrypoint.
  - Smoke assertions updated accordingly.
  - API.md updated to document split between not-ready (`WORLD_PACK_NOT_READY`) and real formatting failure (`CLOCK_FORMAT_ERR`).

### A3. Startup Flow Single Source / 启动流程单一入口

- Problem / 问题:
  - Runtime prerequisites were scattered across dev scripts, CI steps, and manual commands.
- Decision / 决策:
  - Introduce unified server script `prepare:runtime` as the single startup preparation entry.
- Enforcement / 落地:
  - Reused in Linux/Windows start scripts and GitHub Actions smoke workflow.

### A4. Explicit Hard Requirements / 硬性要求显式化

- Decision / 决策:
  - Document non-negotiable startup/testing requirements in README:
    - Run `prepare:runtime` before startup.
    - Degraded startup is acceptable for empty fresh-clone data state.
    - World-pack gated endpoints return `503/WORLD_PACK_NOT_READY` when runtime is not ready.
