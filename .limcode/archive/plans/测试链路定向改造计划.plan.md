# 测试链路定向改造计划

**来源**: `.limcode/review/测试链路重构评估.md`
**总体判断**: 不需要"重构整个测试链路"，需要进行的是**定向、分阶段的补充和改造**

---

## Phase 1 — 建立度量基线（低成本高收益）

### 1.1 启用 coverage 度量
- **文件**: `apps/server/vitest.config.ts`
- **操作**: 添加 `coverage` 配置块
  ```ts
  test: {
    coverage: {
      provider: 'v8',
      reportsDirectory: './coverage',
      reporter: ['text', 'lcov', 'clover'],
      // 初始阈值可设低，后续逐步提升
      thresholds: {
        lines: 60,
        branches: 45,
        functions: 50,
        statements: 60
      }
    }
  }
  ```
- **注意**: 需要安装 `@vitest/coverage-v8` 依赖
- **验证**: `pnpm --filter yidhras-server exec vitest run --config vitest.unit.config.ts --coverage`

### 1.2 CI 中增加覆盖率输出
- **文件**: `.github/workflows/server-tests.yml`
- **操作**: 在 `Run server integration tests` 步骤后/前增加 unit 测试+覆盖率步骤
  ```yaml
  - name: Run server unit tests with coverage
    run: pnpm --filter yidhras-server exec vitest run --config vitest.unit.config.ts --coverage
  ```
- 可选：添加 lcov 上传步骤（如 codecov/codecov-action@v5）

### 1.3 将 web 测试纳入 CI
- **文件**: `.github/workflows/server-tests.yml`（或创建独立 `web-tests.yml`）
- **操作**: 新增 web 单元测试步骤
  ```yaml
  - name: Run web unit tests
    run: pnpm --filter web test:unit
  ```
- **注意**: web 包的依赖只有 vitest + nuxt，不需要 browser 环境，可直接在 node 环境运行

## Phase 2 — 补齐高风险盲区

### 2.1 `ai/registry.ts` 单元测试
- **路径**: `apps/server/src/ai/registry.ts`
- **测试内容**:
  - YAML 文件加载（有效路径、路径不存在、内容语法错误）
  - 多配置合并优先级（default < environment < world pack）
  - 版本协商（version 不匹配时 fallback 行为）
  - 注册表缓存命中/失效

### 2.2 `ai/task_decoder.ts` 单元测试
- **路径**: `apps/server/src/ai/task_decoder.ts`
- **测试内容**:
  - Schema 校验通过/拒绝
  - Alias 解析（短名称→完整配置）
  - Unwrap 行为（嵌套结构展开）
  - 边界：空 schema、缺失字段、多级嵌套

### 2.3 `ai/observability.ts` 单元测试
- **路径**: `apps/server/src/ai/observability.ts`
- **测试内容**:
  - Prisma 写入成功
  - Prisma 写入失败时的降级行为（should not throw）
  - context 为 null 时的 warn 日志输出
  - 调用记录完整性（trace_id、latency、provider）

### 2.4 OpenAI adapter 集成测试
- **路径**: `apps/server/src/ai/providers/openai.ts`
- **测试内容**:
  - Mock HTTP 服务器模拟 OpenAI API 响应
  - 结构化输出（json_schema）请求构建
  - 工具调用（tools/tool_choice）请求构建
  - 错误处理：网络超时、HTTP 4xx/5xx、无效响应格式
  - 流式（SSE）响应处理

### 2.5 HTTP 路由契约测试
- **测试内容**:
  - 请求校验：预期 query params、body shape、content-type
  - 错误响应格式一致性
  - CORS 头检查
  - 速率限制/熔断器响应

### 2.6 `packages/contracts/` 契约验证测试
- **路径**: `packages/contracts/src/`
- **测试内容**:
  - Zod schema 编译的正确性
  - 类型守卫（type guard）对 JSON 数据的验证
  - 多版本兼容性（backwards-compatible schema 变更）

## Phase 3 — 深化浅层测试

### 3.1 Web Pinia store 完整测试
- **目标文件**: `apps/web/stores/*.ts`, `apps/web/features/*/store.ts`
- **测试内容**:
  - State 初始值正确性
  - Actions 执行后 state 预期变更
  - Getters 派生计算的正确性
  - 边界条件：空数据、网络错误、部分更新

### 3.2 拆分巨型测试用例
- **目标文件**:
  - `apps/server/tests/integration/agent-scheduler.spec.ts`（910行，1个 it）
  - `apps/server/tests/unit/context_module.spec.ts`（360行，1个 it）
  - `apps/server/tests/e2e/scheduler-queries.spec.ts`（375行，1个 it）
- **原则**: 每个场景拆分为独立 `it()` 块，保持测试原子性

### 3.3 引入 @vue/test-utils 做组件测试
- **目标**: 关键 UI 组件渲染测试
- **工具**: `@vue/test-utils` + `happy-dom` 或 `jsdom`
- **覆盖**: 
  - 边缘状态（loading、empty、error）
  - Props 产生正确渲染
  - 用户交互触发正确的事件

## Phase 4 — 引入缺失的测试类型

### 4.1 Scheduler 循环 benchmark
- **路径**: `apps/server/tests/bench/scheduler.bench.ts`
- **内容**: 用 vitest `bench` API 测量 `runAgentScheduler()` 在 10/50/100 agents 下的执行时间和创建 job 数

### 4.2 AI inference benchmark
- **路径**: `apps/server/tests/bench/inference.bench.ts`
- **内容**: 测量 `buildAiTaskRequestFromInferenceContext()` 和 `resolveAiRoute()` 的吞吐量

### 4.3 压力测试（k6/artillery）
- **路径**: `apps/server/tests/stress/health.stress.ts`（k6）或 `apps/server/tests/stress/scenarios.yml`（artillery）
- **内容**: 测试 `/api/health`、`/api/status` 在并发请求下的响应时间和错误率

### 4.4 属性测试（fast-check）
- **路径**: `apps/server/tests/unit/prompt_building.pbt.spec.ts`
- **内容**: 用 `fast-check` 生成随机 prompt 配置，验证 `adaptPromptBundleToAiMessages` 的输出始终符合预期结构

---

## 执行顺序

1. **Phase 1**（建立度量基线）→ 必须先做，否则无法量化后续进展
2. **Phase 2**（补齐盲区）+ **FIX**（拆分巨型用例）→ 高安全风险，并行
3. **Phase 3**（深化浅层）→ 依赖 Phase 2 完成后
4. **Phase 4**（新类型测试）→ 长期规划，可并行或延后

---

## 不建议做的事

- ❌ 更换测试框架（vitest 已是正确选择）
- ❌ 推翻 helpers/fixtures 重新设计（现有设计良好）
- ❌ 给每个文件强制写测试（优先高风险模块）
