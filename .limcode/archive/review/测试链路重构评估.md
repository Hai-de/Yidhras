### M2 · 测试质量问题发现

- 状态: 已完成
- 记录时间: 2026-04-25T19:45:43.550Z
- 已审模块: web 测试, server 单元测试, server 集成测试, server e2e 测试
- 摘要:

  ## 发现的核心问题

  ### 问题 1：Web 端测试极度薄弱（严重）

  `apps/web` 有 ~63 个 Vue 组件 + ~100 个 TS 源文件，仅 19 个单元测试。每个测试文件通常只覆盖 1-2 个浅层断言。例如 `graph.store.spec.ts` 整个文件只验证 `autoRefreshMode` 赋值后 state 是否正确——没有任何 store action、getter、边界条件测试。

  **根本原因**：Web 测试仅用 `environment: 'node'`，无浏览器/组件级测试。Vue 组件、composables、Pinia stores 的实际行为几乎没有验证。

  ### 问题 2：单体巨型测试用例（中等）

  多个测试文件包含单个超长 `it()` 块：

  - `agent-scheduler.spec.ts`：1 个测试 910 行，覆盖 10+ 种场景
  - `context_module.spec.ts`：1 个测试 360 行，构建全套 stub 后验证数十个断言
  - `scheduler-queries.spec.ts`：1 个测试 375 行，验证 15+ 个 API 端点

  **影响**：一个断言失败即整测试失败，无法定位具体哪个子场景出问题。迭代维护时新增场景只能继续往里塞，恶性循环。

  ### 问题 3：测试覆盖盲区（严重）

  **零覆盖率工具**：项目中没有任何 coverage 配置（c8/istanbul/v8），无法量化缺口。

  **已知盲区**（来自 TODO.md AI 网关审计）：
  - `registry.ts`（YAML 加载/合并）无测试
  - `task_decoder.ts`（schema 校验/alias/unwrap）无测试
  - `observability.ts`（Prisma 写入）无测试
  - OpenAI adapter 无集成测试
  - HTTP 路由契约无测试

  **额外盲区**：
  - `packages/contracts/` 仅有类型定义，无契约测试
  - `domain/` 层多个模块无对应测试
  - Rust sidecar 无测试

  ### 问题 4：缺少测试类型（严重）

  TODO.md 明确要求的：
  - **基准测试**（benchmark）：零
  - **压力测试**（stress）：零
  - **性能回归测试**：零

  应有但缺失的：
  - 快照测试（snapshot）
  - 属性测试/模糊测试（property-based/fuzz）
  - 组件渲染测试（Vue component testing）

### M3 · 最终判断：定向改造，非全量重构

- 状态: 已完成
- 记录时间: 2026-04-25T19:46:04.214Z
- 摘要:

  ## 判断结论

  **不需要"重构整个测试链路"。需要进行的是定向、分阶段的补充和改造。**

  ### 为什么不需要全量重构

  现有测试基础设施的核心骨架——vitest workspace 多项目配置、三层测试分层（unit/integration/e2e）、隔离环境（`createIsolatedRuntimeEnvironment`）、信封断言工具集——设计合理、结构清晰。推翻重来的成本远高于在其上增量建设。

  真正的问题是 **"写完就忘"** 的增量退化——代码持续演进但测试没有跟上。这不是架构问题，是流程问题。

  ### 应做的事（按优先级）

  **Phase 1 — 建立度量基线（低成本高收益）**
  1. 在 `vitest.config.ts` 中添加 `coverage: { provider: 'v8' }`
  2. CI 中增加覆盖率报告输出，设定初始阈值（如 lines 60%）
  3. 补充 web 测试到 CI 流程

  **Phase 2 — 补齐高风险盲区**
  1. AI gateway 模块：补充 `registry.ts`、`task_decoder.ts`、`observability.ts` 单元测试
  2. 补充 OpenAI adapter 集成测试
  3. 补充 `packages/contracts/` 契约验证测试

  **Phase 3 — 深化浅层测试**
  1. Web 端：用 `@vue/test-utils` 补充 Pinia store 测试（actions/getters/边界）
  2. 拆分巨型测试用例（`agent-scheduler.spec.ts`、`context_module.spec.ts` 等）
  3. 补充关键 composable 测试

  **Phase 4 — 引入缺失的测试类型**
  1. 用 vitest `bench` API 为 scheduler 循环、AI inference、memory compaction 建立 benchmark
  2. 引入压力测试框架（如 k6/artillery）测试 HTTP 端点极限负载
  3. 引入属性测试（如 fast-check）用于 prompt 构建和 schema 校验

  ### 不建议做的事

  - ❌ 不要更换测试框架（vitest 已是正确选择）
  - ❌ 不要推翻 helpers/fixtures 重新设计（现有设计良好）
  - ❌ 不要给每个文件强制写测试（优先高风险模块）
- 结论:
