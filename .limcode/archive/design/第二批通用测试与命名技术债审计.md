# 第二批通用测试与命名技术债审计

## 背景

第一批强依赖 active `death_note` 的 e2e 已完成显式化，并且 6 个目标文件全部通过。当前进入 Phase D，需要把剩余测试中的 `death_note` 默认化依赖拆成三类：

1. **可直接迁移为通用框架测试**
2. **仍应保留为 `death_note` 场景测试**
3. **表面上含有 `death_note` 文本，但实质是 fixture / 内容语义，不应误判为默认 active-pack 技术债**

---

## 结论摘要

- `apps/server/tests/e2e/` 中已经有一批文件没有显式 `activePackRef`，但其断言多数是通用 runtime / scheduler / operator 语义，适合作为**第二批通用 e2e 候选**。
- 也有一些 e2e 虽然没有 helper 显式 active-pack，但已经直接访问 `world-death-note` 路由或依赖 `death_note` 内容，属于**混合型/待拆分**，不能简单做字符串替换。
- unit / integration 层存在大量 `world-death-note`、`death_note`、`available_world_packs: ['world-death-note']`、`pack_id: 'world-death-note'` 固定值；其中一部分是**默认化命名技术债**，另一部分是**合法的场景包 fixture**。

## 二、混合型 e2e：不能直接视为“通用测试”

这些文件虽然可能没有完全显式 active-pack，或者仍带 `death_note` 路由/文本，但不适合直接粗暴改成通用 pack。

1. `apps/server/tests/e2e/smoke-endpoints.spec.ts`
   - 包含通用 smoke 内容
   - 但也直接请求：
     - `/api/packs/world-death-note/projections/timeline`
   - **判定**：混合型
   - **建议**：拆成
     - 通用 smoke endpoint 测试
     - pack projection scenario 测试

2. `apps/server/tests/e2e/experimental-plugin-runtime-web.spec.ts`
   - 直接访问 `/api/experimental/runtime/packs/death_note/load`
   - **判定**：实验 runtime + 场景 pack 混合
   - **建议**：明确保留 `death_note` 作为 scenario pack，或拆分 registry 行为与 pack load 行为

3. `apps/server/tests/e2e/experimental-runtime.spec.ts`
   - 默认禁用分支更通用
   - 启用后分支直接 load `death_note`
   - **判定**：一半通用、一半场景
   - **建议**：拆分为
     - operator API default-disabled 通用测试
     - explicit `death_note` experimental runtime load 测试

## 四、命名技术债审计

### 4.1 明确属于“默认化命名债”的模式

以下模式值得在第二轮中逐步治理：

1. `available_world_packs: ['world-death-note']`
   - 见于多处 unit/runtime/context 测试
   - 如果测试只是验证数组结构/状态聚合，则应改成从 fixture 的 pack metadata 派生，而不是硬编码 death-note

2. `pack_id: 'world-death-note'`
   - 见于 memory/context/runtime sidecar 相关 unit 测试
   - 若测试目的是 pack-aware framework 行为，建议抽为共享测试常量或通过 fixture metadata 注入

3. `sim.init('death_note')` / `pack_ref: 'death_note'`
   - 在若干 integration world-engine sidecar 测试中仍直接出现
   - 若这些是 world-engine integration 通用验证，应考虑迁到 `example_pack`
   - 若它们依赖 `death_note` 内容结构，则应明确标记为 scenario integration

### 4.2 不应误判为命名债的模式

以下出现 `death_note` 文本，不应机械治理：

1. `invoke.execute_death_note`、`invoke.claim_death_note`
   - 这是场景 capability，不是默认 active-pack 假设

2. `death_note_*_v1` prompt preset
   - 这是场景包 AI 配置
   - 只有当测试目标是“框架通用 gateway 行为”时才考虑抽象；否则不应去名

3. `death_note.README.md` asset path
   - 在 plugin runtime web 测试中，这更像 pack asset fixture
   - 它是**内容命名**，不是**active-pack 默认化**

## 五、建议的第二批执行顺序

### Phase D1：通用 e2e 显式化

按以下顺序推进：

1. `smoke-startup.spec.ts`
2. `overview-summary.spec.ts`
3. `scheduler-runtime-status.spec.ts`
4. `scheduler-queries.spec.ts`
5. `access-policy-contracts.spec.ts`

目标：

- 全部改为显式 `activePackRef: 'example_pack'`
- 必要时配套 `seededPackRefs: ['example_pack']`
- 彻底摆脱 helper 兼容默认 `['death_note']`

### Phase D2：混合型 e2e 拆分

优先处理：

1. `smoke-endpoints.spec.ts`
2. `experimental-runtime.spec.ts`
3. `experimental-plugin-runtime-web.spec.ts`

目标：

- 把“通用框架断言”和“scenario pack 断言”拆开
- 避免一个文件同时承担 generic + death_note scenario 两种职责

### Phase D3：unit/integration 命名债清理

优先关注目录：

- `apps/server/tests/unit/runtime/**`
- `apps/server/tests/unit/context*`
- `apps/server/tests/unit/memory_*`
- `apps/server/tests/integration/world_engine_*`

治理策略：

- 优先把固定 `world-death-note` 替换为 fixture metadata 派生值
- 不对真正的 scenario capability / prompt / asset 名称做误伤性去名

## 六、风险

1. **把 scenario fixture 误当 generic debt 清掉**
   - 会损坏 `death_note` 参考包覆盖面

2. **把 generic test 继续留在兼容默认模式**
   - 会拖慢未来移除 `DEFAULT_SEEDED_PACK_REFS = ['death_note']`

3. **混合型文件不拆分只补常量**
   - 会继续保留职责混叠，后续维护成本高

---

## 七、审计结论

当前最合理的下一步不是再改 helper，而是：

1. 先完成第二批真正 generic 的 e2e 显式化
2. 再拆混合型 e2e
3. 最后处理 unit/integration 命名债

换句话说，下一阶段的核心不再是“让 `death_note` 不当默认”，而是：

> 让测试清楚表达自己是在验证 framework，还是在验证 `death_note` 这个 reference/scenario pack。
