# 增强功能储备

> 本文件记录经评估后确认有价值但当前阶段不适合实施的功能增强。
> 每项标注判定时间和暂缓原因。

---

## 插件 per-operator 配置

- **提出时间**: 2026-04-28
- **暂缓原因**: 当前系统选择 pack 作为插件管理的最小粒度，此设计是刻意的架构决策而非遗漏。per-operator 插件配置会导致同一 pack 内 operator 看到不同的插件行为，与"pack 作为世界状态单元"的核心假设冲突。需要在此之前先澄清"插件的归属权"和 P2P agent 自主行为时的权限传递模型。
- **前置条件**:
  1. 明确插件生命周期中的"所有者"概念（pack vs operator vs agent）
  2. 解决 agent 自主推理时的插件配置来源问题
  3. 评估是否需要比 pack 更细的"租户"抽象
- **替代方案**: 通过 capability 控制 operator 的插件管理权限（已有 `MANAGE_PLUGINS`），无权限的 operator 无法管理插件，但不影响已激活插件的运行时行为

## Streaming/SSE 支持（AI 推理流式响应）

- **评估时间**: 2026-05-01
- **暂缓原因**: 当前架构下流式传输没有受益方。全链路（OpenAI 适配器 → 网关 → 工具循环 → 推理服务 → HTTP 路由 → 前端）均为同步请求-响应模式，无任何流式能力。
  - **推理由仿真循环异步批量驱动**，非用户交互触发。AI 调用发生在 `job_runner.ts` 的后台 worker 中，没有 HTTP 客户端在等待响应。流式响应需要一个实时消费端，但作业执行者是后台进程而非前端用户。
  - **前端是只读轮询控制台**，不触发推理。所有推理数据通过 3-30 秒间隔的 fetch 拉取已完成的作业结果，无可展示流式文本的 UI 组件。
  - 工具循环 (`tool_loop_runner.ts`) 的串行阻塞模型与流式增量解析存在根本性架构冲突，改动波及面过大。
- **此评估的时效性**: 此结论基于 2026-05-01 的代码状态。如果以下任一前置条件发生实质性变化，此评估自动过期，应重新分析：
  1. 前端新增操作者交互式触发推理的功能（非仅查看已完成作业）
  2. 引入长文本生成场景（如 narrative generation，单次输出需要数秒以上）
  3. 引入交互式 tool call 审批流程（操作者在 UI 中实时审查并批准 AI 的工具调用）
  4. 推理入口从纯仿真循环内部改为 HTTP 请求驱动的同步调用
- **当前替代方案**: 如果仅关注网络层的可观测性改善（区分连接失败和响应解析失败），最低成本切入点是仅在 OpenAI 适配器 (`providers/openai.ts`) 内部加 `stream: true` + 本地缓冲组装，不改任何上游接口。但此改动仅改善错误诊断精度，不带来用户体验提升。
