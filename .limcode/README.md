# .limcode 资产分层说明

`.limcode/` 用于存放项目的过程资产，而不是长期稳定事实源。

稳定事实优先看：

- `README.md`
- `docs/INDEX.md`
- `docs/API.md`
- `docs/ARCH.md`
- `docs/LOGIC.md`
- `docs/capabilities/*`

## 当前分层

### 1. 活跃 / 参考过程资产

- `design/`
  - 当前仍在推进，或仍值得直接参考的设计文档
- `plans/`
  - 当前仍在推进，或仍值得直接参考的执行计划
- `review/`
  - 当前仍在推进，或仍值得直接参考的评审记录

这里不要求所有文件都属于“当前主线”，但应至少满足以下之一：

- 当前 `progress.md` 仍直接引用
- 稳定文档或后续工作仍会直接引用
- 主题尚未完全退出活跃讨论范围

### 2. 归档过程资产

- `archive/design/`
- `archive/plans/`
- `archive/review/`

用于存放**已完成实施**、但仍保留历史索引价值的过程资产。

这类文件通常满足：

- 对应里程碑/计划已完成
- 相关实现已进入代码与稳定文档
- 继续留在根层会造成活跃层噪音

### 3. 历史草案

- `archive/historical/design/`

用于存放**早期草案、已被后续设计吸收、或文档自身已不再代表当前现状**的设计文件。

这类文件保留的目的是追溯，不是当前决策依据。

## 使用规则

### 新文档放哪里

- 还在推进的新设计/计划/评审：放 `design/`、`plans/`、`review/`
- 已完成并退出活跃层：迁入 `archive/`
- 明确属于历史草案或已被替代：迁入 `archive/historical/`

### 更新规则

- 如果一个主题已经在 `docs/*` 中形成稳定事实源，不要继续让 `.limcode` 文档承担主事实源职责
- 如果一个计划已经完成，不要长期留在根层伪装成活跃计划
- 如果一个设计文档主要依赖“当前/本轮/后续”语气，说明它更适合作为过程资产而不是稳定文档

## 判断优先级

如果多个地方说法不一致，优先级如下：

1. 代码与测试
2. `docs/*` 稳定文档
3. `.limcode/progress.md`
4. `.limcode/design|plans|review/` 活跃过程资产
5. `.limcode/archive/` 历史归档
